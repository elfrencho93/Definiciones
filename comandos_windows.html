10 Comandos de CMD de Windows


Comando Descripción Ejemplo de Uso
git init Es un comando que puede crear un nuevo repositorio. Es el primer comando para iniciar un nuevo proyecto en un repositorio de GitHub. $ git init <nombre de repositorio>
git clone Crea una copia de trabajo local del código fuente desde un repositorio remoto, El código se descarga en la máquina automáticamente cuando clonar un repositorio. $ git clone <git-repo-url>
git branch Agrega una nueva rama a una existente, ve todas las ramas existentes y elimina una rama. Crear una rama localmente:
$ git branch <nombre-de-rama>

Una lista de las ramas locales y remotos:
$ git branch -a

Ver todas las ramas y ver quien rama estas trabajando:
$ git branch

% git branch --list

Eliminar una rama:
$ git branch -d <nombre-de-rama>

Cambiar el nombre de la rama:
$ git branch -m <nombre-de-rama> <nuevo-nombre-de-rama>
git checkout Podemos cambiar una rama existente o crear y cambiar a una nueva. Para hacer esto, la rama a la que desea cambiar debe estar presente en su sistema local y los cambios en su rama actual deben confirmarse antes de cambiar. También se puede utilizar para comprobar los archivos. Cambiar a la rama existente:
$ git checkout <nombre-de-ramal>

Crear y cambiar a la rama nueva:
git checkout -b <nuevo-nombre-de-rama>
git add Este comando agrega sus cambios en un archivo al área de preparación para comprobar su versión local con la versión en el repositorio remoto.

Debe agregarse al área de preparación usando el comando git add antes de confirmar un archivo nuevo o modificado.		 Añadir archivos específicos:
$ git add <nombre-de-archivo-1> <nombre-de-archivo-2>

Añadir todos los archivos nuevos, modificados y eliminados:
$ git add -A

Añadir archivos modificados y eliminados:
$ git add -u
git commit Guarda los cambios en su repositorio local. Cada vez que confirme los cambios de su código, debe incluir una breve descripción de los cambios realizados. El mensaje ayuda a otros a comprender los cambios que se han realizado. Confirme cualquier archivo agregado con el comando git add y los archivos que se hayan modificado desde entonces:
$ git commit -a

Confirmar archivos con un mensaje:
$ git commit -m "<mensaje-confirmado>"

Puede reemplazar los dos comandos anteriores con el siguiente comando único:
$ git commit -am "<mensaje-confirmado>"

Modifique la última confirmación con los últimos cambios como una nueva confirmación:
$ git commit -amend -m "<mensaje-confirmado>"
git push Empuja los cambios de archivo comprometidos desde el repositorio local al repositorio remoto para que otros puedan usarlos. También creará una rama con nombre en el repositorio remoto si no existe. $ git push

$ git push <remoto> <nombre-de-rama>

Si su rama es una recién creada, entonces necesita empujar la rama usando uno de los siguientes comandos:
$ git push --set-upstream <remoto> <nombre-de-rama>

$ git push -u origin <nombre-de-rama>
git pull Obtiene los últimos cambios cargados desde el servidor remoto al repositorio local para que pueda tener las últimas actualizaciones de sus compañeros. $ git pull

$ git pull <remoto>

$ git pull <remoto> <nombre-de-rama>
git merge El comando mezcla su rama con su rama principal. La rama principal puede ser una rama maestra (master) o de desarrollo según su flujo de trabajo.

Crea automáticamente una nueva confirmación si no hay ningún conflicto. Antes de ejecutar este comando, debe estar en la rama específica que desea mezclar con su rama principal.		 Cambie a la rama en la que desea fusionar todo:
$ git checkout <rama>

Comando para obtener las últimas actualizaciones. Hay 2 opciones:
$ git pull

$ git fetch

Si no hay conflictos al extraer las actualizaciones, puede fusionar su rama con la rama principal.

Fusionar su rama:
$ git merge
git status Esto proporciona una descripción general del estado actual de su repositorio. $ git init <status>

Devuelve la siguiente información sobre su rama:
Su nombre de rama actual.

Si su rama actual está actualizada o no.

Número de confirmaciones que su rama está detrás de su rama remota.

Cambios que se escenifican y se van a comprometer.

Cambios que no están preparados para la confirmación.

Archivos sin seguimiento.
